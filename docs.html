Documentation

Everything you need to know about using LogShield to sanitize secrets from your logs.

Overview

LogShield is a deterministic CLI tool for sanitizing secrets from logs. It reads log data from stdin or files, applies explicit rule-based detection, and writes safe, structure-preserving output to stdout.

Design Goals

Deterministic output (same input → same output)
Zero false positives by design
Pipeline-native (STDIN → STDOUT)
Local-only, no network, no telemetry

Non-Goals (Intentional)

No probabilistic / AI-based detection
No log reformatting or restructuring
No cloud service or hosted mode

Installation

Install LogShield globally via npm:

bash
npm install -g logshield-cli

Verify installation:

bash
logshield --version

Basic Usage

Scan a log file:

bash
logshield scan app.log > safe.log

Pipe logs via stdin:

bash
cat app.log | logshield scan > safe.log
Auto-detection STDIN is auto-detected when input is piped. No additional flags required.

Input & Output Behavior

Input Sources

LogShield supports two input modes:

1. File input

bash
logshield scan app.log

2. STDIN (pipe)

bash
cat app.log | logshield scan

When data is piped, LogShield automatically switches to stdin mode.

The --stdin Flag

The --stdin flag exists as an explicit override:

bash
logshield scan --stdin

Use cases:

  • When stdin is not auto-detected by the shell
  • When integrating with non-standard runners

In normal usage, --stdin is not required.

Output Guarantees

  • Output is written only to stdout
  • Original line order is preserved
  • Whitespace and structure are preserved
  • Only detected secrets are replaced

Example transformation:

Before
Authorization: Bearer sk_live_123
After
Authorization: Bearer [REDACTED:STRIPE_SECRET]

Flags Reference

Flag Description
scan Scan input logs for secrets
--strict Enable extended detection rules
--stdin Force stdin mode
--json Output machine-readable JSON summary
--summary Print detection summary to stderr
--fail-on-detect Exit with non-zero code if secrets are found
--version Print version
--help Show help

Exit Codes

LogShield uses stable, CI-safe exit codes:

Code Meaning
0 Success, no secrets detected
1 Secrets detected (with --fail-on-detect)
2 Invalid arguments or usage
3 Runtime error

This allows LogShield to be used as a CI gate.

Determinism Guarantees

LogShield guarantees:

  • Same input produces identical output
  • No randomness
  • No heuristic scoring
  • No context-dependent behavior

This makes LogShield safe for:

  • CI pipelines
  • Auditing
  • Diff-based workflows
  • Reproducible builds

Security Model

Local-only by design LogShield runs fully locally. No outbound network calls. No telemetry. No data persistence.

All detection rules are:

  • Explicit
  • Inspectable
  • Versioned

You can audit every rule applied to your logs by reviewing the source code.

Supported Secret Types

LogShield detects common secret patterns including:

  • Stripe secret keys
  • AWS access keys
  • GitHub tokens
  • Authorization Bearer tokens
  • Generic API keys
  • Database connection strings

See the source code for the full rule list.
